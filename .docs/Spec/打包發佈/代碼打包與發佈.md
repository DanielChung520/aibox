## 概覽

| 項目 | 說明 |
|---|---|
| 應用名稱 | ABC 管理系統 |
| 打包框架 | Tauri 2.x |
| 發佈方式 | **GitHub Actions CI（全自動）** |
| 觸發條件 | 推送 `v*` tag 或手動 `workflow_dispatch` |
| 發佈位置 | GitHub Releases + `latest.json` |
| macOS 產出 | `.app.tar.gz`（**非 DMG**，見下方說明） |
| Windows 產出 | `.nsis.zip`（NSIS 安裝檔） |
| 三平台支援 | macOS ARM (aarch64)、macOS Intel (x64)、Windows x64 |

### macOS 為什麼是 `.app.tar.gz` 不是 `.dmg`？

Tauri 在 macOS 上可以輸出 `.dmg` 安裝檔，但 **`.dmg 只能在 Apple Silicon 機器上產生`**。CI 的 macOS runner 是 Intel 機器，即使交叉編譯 Rust 成功，也無法執行 `hdiutil` 製作 DMG。

因此 CI 選擇輸出 `.app.tar.gz`（壓縮過的 App Bundle）：
- 使用者下載後雙擊解壓縮 → 拖入 `Applications` 即可
- 與 DMG 等效，且 CI 可完整自動化

若需要正式 `.dmg`，需在 Apple Silicon 機器上手動打包。

## CI 自動發佈流程

### 觸發方式

#### 方式一：推送 Tag（推薦）

```bash
# 更新版本號（三個檔案同步）
# src-tauri/tauri.conf.json → version
# package.json → version
# src-tauri/Cargo.toml → version

git add .
git commit -m "chore: bump version to x.y.z"
git tag vx.y.z
git push origin main --tags
```

#### 方式二：手動觸發

1. 前往 GitHub → **Actions** 頁面
2. 選 **Build & Release Tauri App**
3. 點右側 **Run workflow** → **Run workflow**

### CI 工作流程

```
push tag v*  或  workflow_dispatch
        │
        ▼
┌─────────────────────────────────────────┐
│  build-macos (aarch64-apple-darwin)     │
│  - npm ci                               │
│  - npm run build                        │
│  - tauri build --target aarch64-apple-darwin
│  - 上傳: .app.tar.gz + .sig             │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────────────────────────────┐
│  build-macos (x86_64-apple-darwin)      │
│  - npm ci                               │
│  - npm run build                        │
│  - tauri build --target x86_64-apple-darwin
│  - 上傳: .app.tar.gz + .sig             │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────────────────────────────┐
│  build-windows (x86_64-pc-windows-msvc)  │
│  - npm ci                               │
│  - npm run build                        │
│  - tauri build                         │
│  - 上傳: .nsis.zip + .sig              │
└─────────────────┬───────────────────────┘
                  │
        ┌────────┴────────┐
        │  release job     │
        │  (needs both)     │
        ├──────────────────┤
        │ - 下載所有 artifact│
        │ - 計算 SHA256      │
        │ - 產生 latest.json│
        │ - 上傳至 GitHub Release│
        └──────────────────┘
```

### GitHub Release 產出

觸發後約 15~25 分鐘，GitHub Release 頁面會出現以下檔案：

| 檔案 | 平台 | 說明 |

| `ABC管理系统_x.y.z_aarch64.app.tar.gz` | macOS ARM | App Bundle（壓縮檔） |
| `ABC管理系统_x.y.z_aarch64.app.tar.gz.sig` | macOS ARM | 簽名檔 |
| `ABC管理系统_x.y.z_x64.app.tar.gz` | macOS Intel | App Bundle（壓縮檔） |

| `ABC管理系统_x.y.z_x64-setup.exe` | Windows | NSIS 安裝檔 |
| `ABC管理系统_x.y.z_x64-setup.exe.sig` | Windows | 簽名檔 |

## 手動本地打包

本地打包僅用於開發調試，正式發佈請使用 CI。

### 前置條件

```bash
# 確認 Rust target 已安裝
rustup target list --installed

# 需包含：
#   aarch64-apple-darwin
#   x86_64-apple-darwin
#   x86_64-pc-windows-msvc  (Windows 需在 Windows 上執行)

rustup target add aarch64-apple-darwin x86_64-apple-darwin
```

### 設定環境變數

```bash
# 私鑰內容（從 ~/.aibox-updater-key 讀取）
export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/.aibox-updater-key)"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD="abc-updater-2026"
```

### 打包指令

```bash
# macOS ARM
npx tauri build --target aarch64-apple-darwin

# macOS Intel
npx tauri build --target x86_64-apple-darwin

# Windows（需在 Windows 環境）
npx tauri build
```

### 本地產出路徑

```
src-tauri/target/{target}/release/bundle/
├── macos/
│   ├── ABC管理系统.app
│   ├── ABC管理系统.app.tar.gz      ← macOS 產出
│   └── ABC管理系统.app.tar.gz.sig  ← 簽名檔
└── nsis/
    ├── ABC管理系统_x.y.z_x64-setup.exe  ← Windows 產出
    └── ABC管理系统_x.y.z_x64-setup.exe.sig
```

## Updater 簽署金鑰設定

### 金鑰說明

Updater 每次發佈時對安裝包簽名，使用者端自動驗證簽名後才允許更新。金鑰一旦更換，所有使用者都需要重新安裝，因此**請勿更換金鑰**。

> **首次設定只需執行一次。** 若已產生過金鑰，直接跳到「確認 GitHub Secrets」章節。

### 1. 產生金鑰對（Tauri CLI 原生格式）

```bash
npx @tauri-apps/cli@2.10.1 signer generate --ci -w /tmp/aibox-updater.key -p "abc-updater-2026"
```

- `-w`：輸出私鑰檔路徑
- `-p`：密碼（用於解密私鑰，**非**金鑰保護密碼）
- 產出：
  - `/tmp/aibox-updater.key`（348 bytes base64 編碼的加密私鑰）
  - 公鑰直接顯示在終端

### 2. 儲存私鑰

```bash
# 複製到安全位置
cp /tmp/aibox-updater.key ~/.aibox-updater-key

# 確認內容（應該是單行 base64，~350 bytes）
wc -c ~/.aibox-updater-key
```

### 3. 更新公鑰到 `tauri.conf.json`

終端輸出會顯示公鑰（格式：`dW50cnVzdGVkIGNvbW1lbnQ6IG1pbmlzaWduIHB1YmxpYyBrZXk6 ...`），將其設定到：

```json
{
  "plugins": {
    "updater": {
      "pubkey": "dW50cnVzdGVkIGNvbW1lbnQ6IG1pbmlzaWduIHB1YmxpYyBrZXk6 ...",
      "endpoints": [
        "https://github.com/DanielChung520/aibox/releases/latest/download/latest.json"
      ],
      "dialog": true
    }
  }
}
```

### 4. 設定 GitHub Secrets

前往 **GitHub → Settings → Secrets and variables → Actions → New repository secret**：

| Secret 名稱 | 值 |
|---|---|
| `TAURI_SIGNING_PRIVATE_KEY` | `~/.aibox-updater-key` 的完整內容（單行 base64） |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | `abc-updater-2026` |

> **密碼可 hardcode 在 workflow**（`.github/workflows/build-tauri.yml` 中的 `env` 區塊），因為密碼只用於解密金鑰，安全性由私鑰本身保證。私鑰才是需要保護的 secret。

### 5. 確認金鑰有效性

```bash
# 測試本地簽名
export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/.aibox-updater-key)"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD="abc-updater-2026"

npx tauri build --target aarch64-apple-darwin

# 確認簽名檔存在
ls -lh src-tauri/target/aarch64-apple-darwin/release/bundle/macos/*.sig
```

## Updater 更新機制說明

### 工作原理

```
┌──────────────────────────────┐
│  使用者電腦上的 ABC Desktop   │
│                               │
│  App 啟動 → Updater 插件     │
│    ↓ 每 4 小時自動檢查一次  │
│    GET latest.json           │
│    (從 GitHub Release 下載)  │
│    ↓                         │
│  比對版本號                   │
│    ├── 最新 < 安裝版本        │
│    │   → 彈窗提示更新        │
│    │   → 下載新安裝包        │
│    │   → 驗證簽名            │
│    │   → 重啟應用            │
│    └── 最新 == 安裝版本      │
│        → 什麼都不做          │
└──────────────────────────────┘
```

### 完整發佈流程

| 順序 | 動作 | 負責方 |
|---|---|---|
| 1 | 開發者推送 `git tag v1.0.3` | 開發者 |
| 2 | GitHub Actions 自動觸發 | GitHub |
| 3 | 編譯前端、編譯 Rust 殼層（三平台） | CI |
| 4 | 對每個平台的安裝包簽名，產生 `.sig` | CI |
| 5 | 產生 `latest.json`（含簽名、URL、SHA256） | CI |
| 6 | 上傳所有檔案到 GitHub Release | CI |
| 7 | 使用者下次開啟 App → 偵測到新版本 | App |
| 8 | 提示「發現新版本 v1.0.3」→「立即更新」→ 下載 → 重啟 | App |

### 使用者視角

```
首次安裝 v1.0.2（.app.tar.gz 解壓縮）
  ↓
正常使用
  ↓
某天開啟 App → 偵測到 v1.0.3
  → 彈窗：「發現新版本，是否立即更新？」
  → 按「立即更新」
  → 背景下載（約 12 MB）
  → 自動重啟 → 已是最新版本
  （不需要重新下載安裝包，不需要手動處理）
```

### latest.json 格式（Tauri 2.x）

```json
{
  "version": "1.0.3",
  "notes": "Release v1.0.3",
  "pub_date": "2026-03-30T12:00:00Z",
  "platforms": {
    "darwin-aarch64": {
      "signature": "<base64 minisign signature>",
      "url": "https://github.com/DanielChung520/aibox/releases/download/v1.0.3/ABC管理系统_1.0.3_aarch64.app.tar.gz",
      "size": 12582912,
      "checksum": "sha256:abc123..."
    },
    "darwin-x86_64": { ... },
    "windows-x86_64": { ... }
  }
}
```

### 更新時機

- **App 啟動時**：自動檢查一次
- **每 4 小時**：背景定期檢查
- **手動檢查**：可擴充「檢查更新」按鈕觸發

## 常見問題

### Q：CI release job 失敗，回報 `find: 'artifacts': No such file or directory`

表示 `download-artifact` 未找到 artifact。常見原因：

1. **build job 失敗**：檢查 `build-macos` 和 `build-windows` 是否成功
2. **artifact 未上傳**：`upload-artifact` 步驟可能找不到指定的檔案（檢查 CI log 中的 `Debug bundle output` 步驟）
3. **矩陣 job artifact 衝突**：`merge-multiple: false` 導致同名 artifact 被覆蓋，確保每個 build job 的 artifact name 不同（CI 已使用 `-updater` 後綴區分）

### Q：CI 報 `Wrong password for that key` 簽名失敗

確認：
1. GitHub Secret `TAURI_SIGNING_PRIVATE_KEY` 設為私鑰檔的**原始 base64 內容**，不是截斷版本
2. GitHub Secret `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` 設為 `abc-updater-2026`
3. 兩者都不要有額外的換行或空白

### Q：前端代碼變更，需要重新安裝嗎？

**不需要。** 設定好 Updater 後，前端代碼變更時，App 啟動會自動檢測新版本並提示更新，使用者無需手動下載安裝包。見上方 [Updater 更新機制說明](#updater-更新機制說明)。

### Q：Updater 提示「簽名驗證失敗」

可能原因：
1. 公鑰和私鑰不匹配 → 確認 `tauri.conf.json` 的 `pubkey` 與產生金鑰時的公鑰一致
2. GitHub Secrets 中的私鑰內容與本地金鑰不一致

### Q：macOS 首次打開被 Gatekeeper 阻擋

應用未經 Apple 簽署，使用者需右鍵 → 打開 → 確認，或在終端執行：
```bash
xattr -cr /Applications/ABC管理系统.app
```

### Q：金鑰遺失怎麼辦？

若私鑰遺失，**無法再發佈可被舊版用戶驗證的更新**。解決方案：
1. 產生新的金鑰對
2. 更新 `tauri.conf.json` 的公鑰
3. 更新 GitHub Secrets 的私鑰
4. 發佈新版本
5. **所有舊版用戶必須重新安裝**（因為驗證新公鑰會失敗）

因此：**金鑰一旦產生，永久保留。**
