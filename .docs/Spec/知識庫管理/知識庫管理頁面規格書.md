# 知識庫管理頁面規格書

## 1 概述與取代聲明

### 1.1 目的與範圍
本文件定義了 ABC Desktop 系統中「知識庫管理」功能的完整規格，涵蓋前端 UI/UX 交互、全棧 API 契約、ArangoDB 數據模型以及與後端 Pipeline 的整合接口。

### 1.2 取代聲明
本規格書**正式取代**舊版文件 `知識庫獨立管理功能規格.md`。
主要變更點：
- **扁平化文件管理**：移除 `knowledge_folders` 集合，所有文件直接隸屬於 `knowledge_roots`。
- **詳情頁三欄設計**：引入「源文件」、「向量」、「圖譜」三頁籤詳情視圖。
- **Pipeline 整合**：明確定義向量化與圖譜提取的狀態追蹤與整合接口。

### 1.3 數據層級 (Data Hierarchy)
`knowledge_roots` (知識庫) → `domain` (領域) → `major` (專業) → `files` (文件) → `chunks` (分塊) / `entities` (實體) / `relations` (關係)

### 1.4 技術棧摘要
- **前端**: Tauri 2.x + React 18 + TypeScript + Ant Design 6.x
- **後端**: Rust Axum (API Gateway) + Python FastAPI (Knowledge Agent)
- **數據庫**: ArangoDB (元數據) + Qdrant (向量數據)

## 7 Pipeline 整合接口

### 7.1 處理流程 (Flow)
文件上傳 → [分塊 Chunking] → [向量化 Vectorization] → [圖譜提取 Graph Extraction]

### 7.2 狀態機定義 (State Machine)
- **pending**: 初始狀態，等待進入隊列。
- **processing**: 後端正在執行任務（向量計算或圖譜推理）。
- **completed**: 任務成功結束，數據已存入 Qdrant 或 ArangoDB。
- **failed**: 任務失敗，需提供錯誤碼與重試機制。

### 7.3 輪詢接口 (Polling)
前端透過 `GET /api/v1/knowledge/roots/:id` 定時更新 `vector_status` 與 `graph_status`。
當狀態變更為 `completed` 時，觸發右側 Tabs 數據刷新。

### 7.4 Knowledge Agent 整合 (port 8007)
當文件上傳成功後，Rust Gateway 調用 Knowledge Agent 異步觸發 Pipeline。

**Request (Trigger Vectorization)**:
- **POST** `/api/v1/ka/vectorize`
- **Body**: `{ "file_id": string, "root_id": string, "domain": string, "major": string }`
- **Response**: `{ "task_id": string, "status": "queued" }`

**Request (Trigger Graph Extraction)**:
- **POST** `/api/v1/ka/extract-graph`
- **Body**: `{ "file_id": string, "root_id": string }`
- **Response**: `{ "task_id": string, "status": "queued" }`

### 7.5 Qdrant 整合 (port 6333)
向量數據存儲與檢索。

- **Collection Name Convention**: `knowledge_root_{root_id}`
- **Vector Dimension**: 1536 (OpenAI standard)
- **Payload Structure**: `{ "file_id": string, "chunk_id": string, "text": string, "metadata": object }`
- **Query Format**: 
  - **POST** `/collections/{collection_name}/points/search`
  - **Body**: `{ "vector": [...], "limit": 10, "filter": { "must": [{ "key": "file_id", "match": { "value": "..." } }] } }`

## 8 實施優先級與風險

### 8.1 實施階段 (Phases)
- **Phase 1 (基礎)**: 實現 `knowledge_roots` 的 CRUD，完成網格與列表視圖的切換。
- **Phase 2 (文件)**: 實現文件上傳與「源文件」Tab 預覽功能。
- **Phase 3 (AI 整合)**: 接通向量列表展示與圖譜力導向圖渲染。

### 8.2 已知風險與對策
- **大文件處理**: 超長文件向量化可能導致超時。
  - *對策*: 使用異步任務隊列，前端顯示進度條。
- **圖譜複雜度**: 節點過多 (1000+) 時 Canvas 渲染卡頓。
  - *對策*: 引入分層載入 (LOD) 與節點聚合策略。
- **數據一致性**: 刪除知識庫時關聯的 Qdrant 向量需同步清除。
  - *對策*: 實施後端級聯刪除機制，確保無孤立數據。
