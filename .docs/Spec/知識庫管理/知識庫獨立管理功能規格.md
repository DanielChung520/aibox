## 1. 背景與問題陳述

### 1.1 現有架構限制

現有架構中，知識庫上架**綁定任務（task_id）**：

| 面向     | 現有做法              | 問題                                 |
| -------- | --------------------- | ------------------------------------ |
| 上架入口 | 綁定 `*-Agent` 任務 | 知識庫應該是獨立的，不應綁定任務     |
| 文件標記 | 只有 `task_id`      | 無法區分「任務文件」與「知識庫文件」 |
| 權限控制 | 基於 `task_id` 隔離 | 知識庫需要獨立的權限體系             |
| UI 體驗  | 在任務下管理文件      | 知識庫應有獨立的管理介面             |

### 1.2 核心問題

```
當前任務聊天室選擇「經寶物料管理代理」
    → Agent 有 `mm_agent_knowledge` capability
    → KA-Agent 檢索時應該只返回 domain="mm_agent" 的知識
    → **但沒有機制確保這個對應關係**
```

**根本原因**：上架時沒有建立「capability ↔ domain」的明確映射。

### 1.3 解決方案

引入**知識庫類型（file_tree_type）**欄位：

| file_tree_type | 說明                     | 範例                            |
| -------------- | ------------------------ | ------------------------------- |
| `task`       | 任務文件（綁定 task_id） | 用戶上傳到任務的文件            |
| `knowledge`  | 知識庫文件（獨立管理）   | 專業知識文件，綁定 domain/major |

### 2.3 核心差異

| 面向     | 任務文件管理（現有）     | 知識庫管理（新增）                 |
| -------- | ------------------------ | ---------------------------------- |
| 組織方式 | 按 task_id 組織          | 按 domain/major 組織               |
| 文件標記 | `file_tree_type: task` | `file_tree_type: knowledge`      |
| 上傳流程 | 選擇任務，上傳文件       | 選擇/創建目錄，選擇 Ontology，上傳 |
| 權限控制 | 基於任務                 | 基於知識庫目錄                     |
| 檢索過濾 | 不適用                   | 根據 agent capability 自動過濾     |

## 3. 數據模型擴展

### 3.1 file_metadata 新增欄位

```python
class FileMetadata:
    # ... 現有欄位 ...

    # ========== 新增欄位 ==========
    file_tree_type: str = "task"  # "task" | "knowledge"

    # 知識庫專屬欄位（當 file_tree_type == "knowledge" 時生效）
    knowledge_root_id: Optional[str] = None  # 知識庫根目錄 ID
    knowledge_parent_id: Optional[str] = None  # 上層目錄 ID（null = 根目錄）
    ontology_domain: Optional[str] = None  # Ontology Domain（如 "mm_agent"）
    ontology_major: Optional[str] = None    # Ontology Major（如 "spec"）
```

### 3.2 新增 Collection：knowledge_roots

**目的**：管理知識庫的根目錄結構

```python
class KnowledgeRoot:
    _key: str  # root_id（如 "mm_agent", "ka_agent", "default"）
    name: str  # 顯示名稱（如 "MM-Agent 知識庫"）
    description: Optional[str] = None  # 說明
    owner_user_id: str  # 創建用戶
    ontology_domain: str  # 綁定的 Ontology Domain
    ontology_majors: List[str]  # 允許的 Major 清單
    created_at: str
    updated_at: str
```

### 3.3 新增 Collection：knowledge_folders

**目的**：知識庫目錄樹結構

```python
class KnowledgeFolder:
    _key: str  # folder_id（如 "mm_agent_spec_001"）
    root_id: str  # 所屬 KnowledgeRoot._key
    parent_id: Optional[str] = None  # 上層目錄（null = 根目錄）
    name: str  # 目錄名稱
    path: str  # 完整路徑（如 "/mm_agent/spec"）
    ontology_domain: str  # 綁定的 Domain
    ontology_major: str  # 綁定的 Major
    created_by: str
    created_at: str
    updated_at: str
```

### 3.4 數據關係圖

```
knowledge_roots (根目錄)
    │
    ├── knowledge_folders (目錄樹)
    │       │
    │       ├── folder_1 ──→ file_metadata (file_tree_type=knowledge)
    │       │
    │       └── folder_2 ──→ file_metadata (file_tree_type=knowledge)
    │
    └── knowledge_folders
            │
            └── folder_3 ──→ file_metadata (file_tree_type=knowledge)

file_metadata (file_tree_type=knowledge)
    ├── knowledge_root_id → knowledge_roots._key
    ├── knowledge_parent_id → knowledge_folders._key
    ├── ontology_domain
    └── ontology_major
```

## 4. API 設計

### 4.1 新增 API Endpoints

| Method | Endpoint                                  | 說明                             |
| ------ | ----------------------------------------- | -------------------------------- |
| GET    | `/api/v1/knowledge/roots`               | 獲取知識庫根目錄列表             |
| POST   | `/api/v1/knowledge/roots`               | 創建知識庫根目錄                 |
| PUT    | `/api/v1/knowledge/roots/{root_id}`     | 更新根目錄                       |
| DELETE | `/api/v1/knowledge/roots/{root_id}`     | 刪除根目錄                       |
| GET    | `/api/v1/knowledge/folders`             | 獲取目錄樹（可選 root_id）       |
| POST   | `/api/v1/knowledge/folders`             | 創建目錄                         |
| PUT    | `/api/v1/knowledge/folders/{folder_id}` | 更新目錄                         |
| DELETE | `/api/v1/knowledge/folders/{folder_id}` | 刪除目錄                         |
| POST   | `/api/v1/knowledge/upload`              | **知識庫文件上傳**（核心） |
| GET    | `/api/v1/knowledge/files`               | 獲取知識庫文件列表               |
| GET    | `/api/v1/knowledge/files/{file_id}`     | 獲取文件詳情                     |
| DELETE | `/api/v1/knowledge/files/{file_id}`     | 刪除知識庫文件                   |

### 4.2 核心 API：知識庫文件上傳

```
POST /api/v1/knowledge/upload
Content-Type: multipart/form-data
```

**請求參數**：

| 參數                 | 類型   | 必填 | 說明                       |
| -------------------- | ------ | ---- | -------------------------- |
| `files`            | File[] | ✅   | 要上傳的文件               |
| `root_id`          | string | ✅   | 知識庫根目錄 ID            |
| `parent_folder_id` | string | ❌   | 上層目錄 ID（空 = 根目錄） |
| `ontology_domain`  | string | ✅   | Ontology Domain            |
| `ontology_major`   | string | ✅   | Ontology Major             |
| `user_id`          | string | ✅   | 上傳用戶                   |

**響應**：

```json
{
  "success": true,
  "message": "文件上傳成功",
  "data": {
    "uploaded": [
      {
        "file_id": "kb_mm_agent_spec_001",
        "filename": "庫存管理規範.pdf",
        "ontology_domain": "mm_agent",
        "ontology_major": "spec",
        "file_tree_type": "knowledge",
        "knowledge_root_id": "mm_agent",
        "knowledge_parent_id": "mm_agent_spec"
      }
    ],
    "failed": []
  }
}
```

### 4.3 與現有 `/api/v1/files/v2/upload` 的關係

| 面向                | files/v2/upload  | knowledge/upload (新)            |
| ------------------- | ---------------- | -------------------------------- |
| file_tree_type      | `task`         | `knowledge`                    |
| task_id             | 必填             | 不使用                           |
| knowledge_root_id   | 不使用           | 必填                             |
| knowledge_parent_id | 不使用           | 必填                             |
| ontology_domain     | 可選             | **必填**                   |
| ontology_major      | 可選             | **必填**                   |
| pipeline            | 分塊→向量→圖譜 | 分塊→向量→圖譜→**編碼** |
| 生成 knw_code       | 否               | **是**                     |

## 6. 核心邏輯：Agent 能力與知識庫對應

### 6.1 現有問題

```
Agent capability: mm_agent_knowledge
    ↓
Agent 可以訪問哪些知識？
    ↓
**沒有明確對應**
```

### 6.2 解決方案：capability ↔ domain 映射

**方案：Agent Registry 存儲 capability ↔ domain 映射**

```python
# agent_display_configs.capabilities 擴展
{
  "id": "jingbao-mm-1",
  "capabilities": [
    "mm_agent_knowledge:mm_agent",      # 可以訪問 domain=mm_agent 的知識
    "ka_agent_knowledge:ka_agent"       # 可以訪問 domain=ka_agent 的知識（如果需要）
  ]
}
```

### 6.3 檢索時的權限過濾（KA-Agent）

```python
# ka_agent/agent.py 檢索時

def _get_allowed_domains(caller_agent_id: str) -> List[str]:
    """根據 Agent capability 獲取允許訪問的 domains"""
    registry = get_agent_registry()
    agent_info = registry.get_agent_info(caller_agent_id)

    allowed_domains = []
    for cap in agent_info.capabilities:
        if cap.startswith("mm_agent_knowledge:"):
            domain = cap.split(":")[1]
            allowed_domains.append(domain)

    return allowed_domains

# 檢索時過濾
def _query_knowledge_files(self, query: str, caller_agent_id: str):
    allowed_domains = self._get_allowed_domains(caller_agent_id)

    # 只返回這些 domain 的知識
    files = file_metadata_service.list(
        file_tree_type="knowledge",
        domain_in=allowed_domains,
        lifecycle_state="Active"
    )
```

## 9. 風險與注意事項

| 風險                  | 影響           | 緩解措施                         |
| --------------------- | -------------- | -------------------------------- |
| 既有任務文件被誤操作  | 文件丟失       | 知識庫管理與任務文件管理完全隔離 |
| domain/major 配置錯誤 | 知識無法被檢索 | 上傳時必選，格式驗證             |
| 遷移腳本出錯          | 數據不一致     | 遷移前備份，提供回滾腳本         |
